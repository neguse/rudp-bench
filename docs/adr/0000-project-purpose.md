# ADR-0000: プロジェクトの目的

- Status: **Proposed**(最上位。ここが変わると 0001 以下は全部影響を受ける)
- Date: 2026-07-10
- 出典: 過去セッションログ(2026-05-29、dfbc0031)のユーザー発言

## Decision

> 「結局知りたいのって『たとえば音声用途なら』どんぐらい接続数いけるか、
> オーバーヘッドあるか、みたいな」
>
> 「安くなくていいんだけど、再現性あるテストを何度でもできることよな」

本プロジェクトの目的は
**「用途アーキタイプごとに、各 transport 実装が何接続まで品質を保って
捌けるか(とそのオーバーヘッド)を、再現可能な実測で知る」**こと。

用途アーキタイプは既にコード・過去レポートで定義済みのものを正とする
(`orchestrator/report/report.go` の anchor 定義):

| anchor | workload | 想定用途 |
|---|---|---|
| **br** | r20p128(20Hz × 128B、同報) | 音声ルーム / 位置同期ブロードキャスト |
| **vr** | r60p200(60Hz × 200B、同報) | VR ポーズ同期(高頻度・中サイズ) |
| **video** | r20p1000(20Hz × 1000B、同報) | 映像フレーム片 / 大きめ状態同期 |
| synthetic | echo / reliable_echo | 接続数スケールの合成基準 |

- 特定の 1 ゲームの採用選定でも、公開記事でもない。「知る」が目的で、
  結果は表とレポートとして repo に蓄積する
- 再現性は目的の一部(1 回きりの数字は成果でない)。コスト(時間・
  計算資源)は再現性より優先しない

## 非目的

- 全ライブラリを 15% 解像度で横並びランクすること(specs/2026-07-08 の
  スコープ宣言を踏襲)
- gate なしのスループット最大値マイクロベンチ
- ライブラリ本体の改造(upstream 公式ノブ/API + 開示、が対戦資格)

## Consequences

- バトル(ADR-0001)は「アーキタイプ別 capacity 表を作る」ための手段
- 表の行は r20p128 等の生名でなく **br / vr / video / echo のアーキタイプ名**
  で提示する(数字の読み手は用途で引くため)
- 「何のために測るか」を再び見失ったら、この ADR に戻る
