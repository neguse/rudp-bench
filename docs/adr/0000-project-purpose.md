# ADR-0000: プロジェクトの目的

- Status: **Proposed**(最上位。ここが変わると 0001 以下は全部影響を受ける)
- Date: 2026-07-10

## Context

rudp-bench は「RUDP/QUIC 実装を同じ workload で比較する benchmark
harness」(README)としてスタートし、v1 では自作実装(coop_rudp /
apex_rudp)を含む 14 transport を測っていた。v2 で測定器の精度・妥当性を
立て直し、tune-to-plateau とバトル(ADR-0001)まで進んだが、
**この比較を何の意思決定に使うのか**が文書化されていない。
目的が未定義だと、対戦カード(自作実装を入れるか)・要求解像度・
regime の現実性(RTT いくつが「現実的」か)がすべて宙に浮く。

## Decision(推定を含む起案 — 事実誤認があれば修正して Accepted に)

本プロジェクトの目的は:

**「自分のゲーム(リアルタイム・マルチプレイヤー)のサーバー transport を、
監査可能な実測に基づいて選定する」**こと。

- 比較データは選定の根拠として残す(設計ノート・開示チューニング・
  censoring 込みの正直な表 = 後から第三者が検証できる形)
- 既製ライブラリで要求を満たせるならそれを採用する。満たせない場合の
  自作(coop_rudp 系)は、この表を「超えるべき基準」として使う

## 明示的に確認したい推定(2 点)

1. **用途**: 上記「自分のゲームの transport 選定」で正しいか。
   (別候補: 公開する比較記事/データセットが主目的 — その場合は
   再現手順の公開性・rig の中立性への要求が一段上がる)
2. **自作実装(coop_rudp)の扱い**: v1 では echo 3000 conns not_broken の
   最強格だったが **v2 に未移植**で、現バトルに不参加。
   目的が transport 選定なら「coop_rudp を v2 に移植して対戦に入れる」は
   本命の選択肢になる(入れるなら ADR-0003 の transport 軸と
   実行順に追加)

## Consequences

- ADR-0001(バトルの目的)はこの ADR の手段として位置づけ直される
  (Q1-Q3 は選定判断の材料。変更は不要の見込み)
- 「ゲームの要求」(想定同時接続数・同報規模・許容遅延・想定回線)が
  定まっているなら、それが解像度要求とマトリクスの優先順位を決める —
  分かっている範囲で本 ADR に追記していく
