# ADR-0001: バトルの目的

- Status: **Proposed**(承認で Accepted、以後の変更は supersede する新 ADR でのみ)
- Date: 2026-07-10
- 依存: ADR-0000(バトルは ADR-0000 の目的に対する手段)

## Context

各ライブラリの tune-to-plateau 済み「最強実装」を比較する対戦表を作る
(2026-07-10 ユーザー指示が起点)。何をもって「強い」とするかが未定義の
まま測定に入り、workload 構成(broadcast のみ)や遅延条件(RTT20ms)が
後から問題化した。目的を先に固定する。

## Decision

バトルは **ゲームサーバー用途の transport 選定**に答える。具体的には
次の 3 問への capacity(品質ゲート内で維持できる最大同時接続数)番付:

- **Q1 接続数スケール**: 1 サーバーに何接続ぶら下げられるか(echo=synthetic、負荷 O(conns))
- **Q2 用途アーキタイプ別の同報 capacity**: br(音声ルーム)/ vr / video の
  各同報パターンを何接続まで捌けるか(負荷 O(conns²)、ADR-0000 の表が正)
- **Q3 loss 耐性**: 回線が荒れたとき Q1/Q2 がどれだけ残るか(TCP 系と RUDP 系の分水嶺)

遅延条件はプレイヤーの現実的な体感域(RTT 50ms 級)で測る。

## 非目的

- 全ライブラリを 15% 解像度で横並びランクすること(原理的に不可能 —
  specs/2026-07-08 のスコープ宣言を踏襲)
- スループット最大値のマイクロベンチ(gate 無しの数字は扱わない)
- ライブラリ本体の改造(チューニングは upstream 公式ノブ/API の範囲で
  --describe に開示、が対戦資格)

## Consequences

- 対戦表は「shape(Q1/Q2)× regime(Q3)」の複数表になる。単一の総合
  順位は作らない — 用途の問いに紐づかない順位は誤読を生むため
- echo を含まない表(2026-07-10 の RTT20 broadcast 表)は参考記録に降格
