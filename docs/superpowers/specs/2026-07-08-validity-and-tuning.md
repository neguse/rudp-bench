# v2 妥当性インフラと tune-to-plateau(次フェーズ)

2026-07-08 起草。[再生計画](2026-07-07-v2-measurement-recovery.md)は測定の
**精度(再現性)**を回復した(定常 warmup・決定的ロス・故障分離・帰属)。
本 spec はその上に載る**妥当性(validity)**を扱う。根拠は 2026-07-08 の
deep research(方法論・DoE・チューニング統合フレームワーク)。

## 精度と妥当性は別物

再生計画で潰したのは「同じ条件で数字がブレる」= precision。だが
**「安定して低い数字」は「アダプタのバグ」と「ライブラリが本当に遅い」を
区別しない**。これは validity の問題で、再生計画の作業は一切効かない
(直交する)。実例: 2026-07-02 の apex 回帰はアダプタの二重ラップで配送 0%、
全テスト green、再現性は完璧だった。precision は「盤石」と言い、validity は
壊れていた。

## 中心原則: 差を測るな、潰せ / 天井を測れ

1. **バグか実力かは測って晴らせない。tune-to-plateau で削り切る。**
   各アダプタをプロファイル見ながら詰め、**もう速くならない点 = ライブラリの
   本当の底**。詰め切っていない数字は「性能」でなく「下限」(≥X)。詰め切った
   もの同士しか並べない。
2. **ハーネス自身の天井を測る。** ダミー/最小トランスポート(raw_udp)の値が
   このハーネス+プロトコルで出せる上限。どのトランスポートの数字も
   「天井の何%か」で読めば、天井に張り付くやつは "ライブラリでなくハーネスを
   測っている" と即断できる。apex 0% はこの基準線が無かったから通った。
3. **層別フロア階層。** L0(raw UDP ループバック)→ L1(トランスポート)→
   L2(RPC/シリアライズ)のゼロペイロード・エコー往復を先に測り、各層は
   「下の層の床 + 期待オーバーヘッド」に収まるべき。乖離が大きい層に
   実装/設定ミスがある。MagicOnion が遅い時、MessagePack 単体 / grpc-dotnet /
   raw TCP のどれかを切り分けられる。
4. **解像度を宣言し、その上でしか言い切らない。** TCP は loss×負荷で崩れる級
   (10〜100倍)は鉄板。ライブラリ間 15% は解像度の下 = 主張しない。
   解像度は比較ごとに tune-to-plateau で買える(全部はやらない、決定が懸かる
   2〜3 個だけ)。

## 実行順(効果が大きく低リスクな順)

### V1: ハーネス天井の基準線(最優先・安い)
- **servers/raw_udp**(v2 benchkit、最小 UDP エコー/broadcast、信頼性・dedup
  なし)を作る。sweep/canonical に L0 基準線として組み込む。
- report に各 capacity を「raw_udp 比」で併記。天井張り付き = ハーネス律速の
  フラグ。
- **完了条件**: raw_udp が全 anchor で最上位、他 transport がそれ以下に並ぶ。
  raw_udp に張り付く transport があれば、それは harness ceiling を測っている。

### V2: 層別フロア + USE 法カウンタ
- L0/L1/L2 のゼロペイロード echo 往復を測る floor スイート。
- 各 run のメタデータに **コア別 CPU**(平均でなく個別コア)・NIC pps・
  syscall 数(bpftrace)・GC カウンタ(.NET は dotnet-counters)を自動収集。
- farm 帰属を「プロセス CPU ≥85%」から「担当コアの飽和」に精緻化。

### V3: tune-to-plateau(決定が懸かる 2〜3 transport だけ)
- 対象を選ぶ(用途 → 候補ペア)。残りは「≥X・未チューニング」の札のまま。
- 各対象: 破断点でプロファイル(perf/flamegraph)→ アダプタ側の無駄
  (コピー・alloc・ロック・バッファ・直列化)を潰す → 測り直す、を頭打ちまで。
  上流公式ベンチと突き合わせて取りこぼし確認。
- **停止条件**: 追加チューニングで測定値が動かず、単一コア CPU か NIC 帯域が
  飽和。飽和資源が無いのに頭打ち → V1 の harness ceiling を疑う。

### V4: DoE で分散帰属(必要になれば)
- ペイロード×レート×接続数×GSO×ピン×GC の要因計画を反復・ランダム化、
  ANOVA で「プロトコル差 vs ハーネスノイズ」を数値分離。

## スコープの正直な宣言

「14 transport を横並びで 15% ランク」は原理的に不可能(解像度の下)。
本フェーズが目指すのは「決定が懸かる数個を、その決定に必要な解像度まで
tune-to-plateau で詰めて比較」。全部そこそこ測る形は捨てる。成熟した
2 ライブラリが検証後も 15% 以内なら、正直な結論は「差なし、性能以外で選べ」
—— それも結論。
