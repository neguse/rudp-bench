# v2 再生計画: 測定系の欠陥と再設計

2026-07-07 起草。v2 の統計装備(N=3 ブロック・無作為化・IQR)は、測定器自身が作る
ぶれを潰す前に張られており、実験計画法の手順(測定系分析 → 系統要因の除去 →
残余ノイズへの反復)を逆順に踏んでいた。本計画はその是正手順を定める。

## 1. 診断

sentinel(定点再測チェック、26 条件、1 回 13 分)を同一環境で 6 回反復した結果:

- ロスなし・定常状態の 17 条件は 6 回とも完全一致 — **CPU 隔離は機能している**
- ぶれた条件は例外なく「ロス注入あり」か「起動直後の挙動を含む」— **残余のぶれは
  環境ノイズではなく測定方法が自分で作っている**

ぶれの発生源は 3 つ、いずれも方法側:

| # | 発生源 | 機構 | 実測 |
|---|---|---|---|
| D1 | ウォームアップが固定 1s(接続数に無関係) | 接続ストーム直後の非定常区間が計測窓に入る run と入らない run に二分される | enet の配送率が ~17% の確率で 0.99 → 0.80–0.89 に落ちる(ledger #14) |
| D2 | ロス注入が毎 run 独立乱数 + 計測窓が短い | 3%×burst16 ではロス事象が窓内に数十回しか入らず、p99 が最悪数回の引きで決まる | ロス系 probe が 10–20%/条件でぶれる。enet throttle の時定数(ledger #12)がこれを増幅 |
| D3 | 測定インフラの故障が結果に混入 | sampler の read 失敗等が「基準からの漂移」として計上される | sentinel 6 run 中 1 件 |

派生問題: ロス下では client farm(測定対象と同じライブラリで負荷生成)が
ライブラリ側の stall で崩れ、それが「farm 律速」と誤帰属されて censoring される
(ledger #6/#13)。ブロック間で「正直 break 0」と「censored ≥128」に割れる直接原因。

## 2. 資産査定

**生きているもの(再利用する):**

- ハーネスと校正一式: 理論フロアとの一致(1% 双方向ロスで delivery 0.9803 vs
  理論 0.9801)、会計零点、故障注入、netem 実測 gate、v1/v2 の独立系一致
- ロスなし条件の結果全般(ぶれゼロを反復で確認済み)
- boundary の大信号(TCP 系の秒単位崩壊 vs UDP 系のフロア張り付き —
  ぶれの 2 桁上のオーダー差であり方法欠陥では覆らない)
- sentinel 本体(13 分/回)と block/aggregate 基盤(測定系分析の道具として転用)

**死んでいるもの(publish から撤去・置換):**

- capacity @ loss-worst(3%×burst16)全表 — D2 + 誤帰属 censoring の合成。
  そもそも同 regime は absolute 予算がフロア負けして infeasible であり、
  容量主張の意味論が立たない
- 起動直後感受性のあるセル(enet の wired capacity 際 等)の確定値
- 「N=3 で確定」という信頼度表記そのもの(必要反復数は Phase 2 で実測し直す)

## 3. 原則

1. ぶれは統計で吸収する前に発生源で潰す。反復・ブロック化は「潰せないノイズ」専用
2. 測定器の変更(D1–D3 の修正)はプロトコル変更であり、修正後に基準を取り直す
3. イテレーション速度は一級の要件。確定測定の所要時間は「必要反復数を実測で決める」
   ことで支払う(全セル一律 N=3 をやめる)
4. 容量をロス下で主張する場合は、実在する運用点(1% 級)で測る。infeasible な
   角(3%×burst16)は boundary(劣化特性)でのみ扱う

## 4. 計画

> **進捗(2026-07-07)**: Phase 0 / Phase 1 完了(受け入れ実測値は各 Phase 末尾)。
> Phase 2 実行中。

### Phase 0: 衛生処置(半日)

- v2-draft レポートの loss-worst capacity 表と E2 期の stale な単発表に
  「測定系欠陥により無効(本計画参照)」の注記を入れる
- sentinel の起動ラッパーを作る: bench.slice への systemd-run 起動 +
  netns 残留の冪等クリーンアップを内蔵(2026-07-07 に手動で踏んだ罠の恒久化)
- sentinel の結果コードを INVALID(測定不成立)/ DRIFT(測定成立・基準乖離)に分離
- **完了条件**: 1 コマンドで sentinel が回り、環境不成立が漂移と別コードで報告される

### Phase 1: 系統ぶれの除去(数日)

1. **D1: ウォームアップの定常判定化** — 固定 1s をやめ、送受レートと delivery の
   安定を検出してから計測窓を開く(上限付き)。
   検証: enet の二峰再現条件(wired br/video の capacity 際)を 20 回反復し、
   全 run が単一モードに収束すること
2. **D2: ロスの固定パターン注入** — netem の独立乱数をやめ、事前生成した
   drop trace(seed 付き)を全 run・全 transport に再生する。netem は seed を
   持てないため、パケット連番ベースで落とす自前機構(tc/eBPF または中継)を作る。
   検証: 同一 trace で同一 transport を 10 回反復し、staleness p99 の変動係数が
   ロスなし条件と同水準まで縮むこと
3. **D2 補完: 事象アライン指標** — 窓全体の p99 に加え「ロス事象ごとの回復時間」を
   出力する(事象数を制御できるため推定が安定する)
4. **D3: 故障の分離** — infra 故障(sampler 等)は run を INVALID にして統計から
   除外・別集計
5. **farm 誤帰属の解消(ledger #6/#13)** — client の CPU 実測を使い
   「farm の計算資源律速」と「ライブラリ client 側の stall」を別ラベルにする。
   ロス下 capacity を 1% 級に移す(下記 Phase 3)なら発火頻度自体が下がる見込み

**Phase 1 の実測結果(受け入れ基準の判定)**:

- **D1(定常判定つき warmup、benchspec v2)**: enet の二峰再現条件
  (wired br c101、従来の bad mode 発生率 ~17%)で 20 反復 ×3 系列を実測。
  - steady 検出のみ(2.76s で開窓): bad mode 再発(0.68 / 0.85)—
    throttle の過渡はレートが数秒 flat に見えた後 ~13s まで残ると確定
  - 固定 15s warmup: 20/20 pass(min 0.983)
  - **steady + 最小 warmup 15s(採用構成): 20/20 が delivery 0.9975–0.9985
    の単峰(幅 0.001)** — 受け入れ達成
  - 定常判定の予定式は「250ms バケット単体の ±10%」では送信 slot 境界との
    エイリアシング(バケット値が 52/65/78/91 と量子化振動)で恒久不成立に
    なるため、「連続2つの 1 秒窓の合計 ±10% + 期待レート下限 0.9」に変更済み
- **D2(決定的 loss 注入、eBPF/TCX + 事前生成 trace)**: msquic r60p200 c4、
  3%×burst16、60s × 10 反復。
  - staleness p99 の CoV: 乱数 netem **26.0%** → 固定 trace **4.0%**
    (ロスなし基準 2.0%、ヒストグラム bin 1 段 ≈ 4.5% なので量子化幅と同水準)
    — 受け入れ達成
  - ping 実測でも trace 先頭 400bit の drop 数(14)とビット単位で一致
  - 副産物: **事象アライン指標 update_gap p99 は乱数 loss でも CoV 0.0%**
    (事象数でなく事象ごとの回復を測るため)。boundary 主張の一次指標候補
- **D3 / farm 誤帰属**: sentinel は INVALID(測定不成立)と DRIFT を分離
  (終了コード 4 / 3)+ flap 政策(乖離の即時再測、2 回連続でのみ報知)。
  sweep は invalid 点を 1 回再測し、再発時は measurement_invalid として
  censored 化(break に数えない)。farm censoring は「client farm CPU 実測が
  割当の 85% 以上」のときのみ。CPU が暇で sched latency 超過は client_stall
  として正直に break + 開示(旧実装は end-to-end latency を farm 帰属に
  使っており、これが loss regime の `!` 食い違いの主因だった)

### Phase 2: 測定系分析(マシン時間 ~1 日)

- 修正後の測定器で代表セル(wired / 1% loss × UDP 系 / TCP 系 × anchor 3 種)を
  各 10–20 回反復し、セル型ごとの分散を実測する
- そこから **セル型 → 必要反復数** の表を確定する(見込み: 大半 N=1、
  gate 近傍のみ N=2–3)
- sentinel の基準値をこの反復群の中央値に差し替える(ledger #15)。
  報知条件は「2 回連続乖離」とする(単発ぶれの残余対策)
- **完了条件**: 必要反復数の表が本ドキュメントに追記され、sentinel が新基準で
  連続 5 回、偽報知ゼロで通ること

**Phase 2 の実測結果(2026-07-07、修正後の測定器で 2 regime × 3 transport ×
3 anchor × 10 反復 = 180 run)**:

18 セル中 16 セルで staleness p99 の CoV ≤ 6.7%(大半 ≤4% = ヒストグラム
bin 量子化幅)。delivery の CoV は全セル ≤3%(外れセル除く)。

| セル型 | run 間ばらつき | 必要反復数 |
|---|---|---|
| 容量点から離れた点(wired / loss とも、UDP 系・TCP 系とも) | CoV ≤ ~5%(bin 量子化水準) | **N=1** |
| gate 際(容量の ~0.9 倍より上) | 通常は緊密だが、稀な崩壊モードが乗る(msquic wired br c128: 9/10 が 74–82ms、1/10 が 4.7s/delivery 0.53) | **N=3 の median**(capacity 確定・sentinel 基準はここに該当) |
| TCP 系 × loss | drop 位置を固定しても回復動態自体が確率的(websocket vr 1%: 344ms–5.2s)— 測定器でなく transport 固有の分散 | **N≥3 で median [IQR] 表記**。事象アライン指標(update_gap)の併記を検討 |
| enet(最小 warmup 15s 宣言下) | 全セル CoV ≤6.7% — 二峰性は解消済み | N=1(gate 際は上記どおり) |

補足: TCP 系 × loss で定常未達(cap fallback)になるのは「nominal rate に
届かない」という開示であり正しい挙動(websocket video 1% で 0/10 が steady)。
update_gap は当初 class 混合で実装され must-deliver の送信間隔(1Hz=1s)が
p99 を支配していた(全セルで p99≈1016ms に張り付く)ため class 別に分離した。
分離後の loss_tolerant gap のばらつきは Phase 3 のデータで確立する。

### Phase 3: 主張の再測(マシン時間 ~1–2 日)

- capacity @ wired(anchor 3 セル)— 再測、探索は旧 break 点からの warm-start
- capacity @ 1%×b1–b4(新設)— loss-worst capacity の置換。実在する運用点
  (国際回線・モバイル級)での収容数主張
- boundary — 境界近傍(1% 帯)+ 両端に細分を絞る。q25 負荷段は廃止
  (floor と q75 で挟む)
- レポートの全表を新測定器の値で再生成
- **完了条件**: ブロック間食い違いフラグ(`!`)がゼロ、全表が新測定器由来

### Phase 4: 継続運用

- sentinel を変更ごと(および日次 cron)に実行、基準は Phase 2/3 の集約値
- 測定器・アダプタに触れた変更は sentinel 通過を merge 条件にする
  (2026-07-02 の v1 回帰 — アダプタ改修で apex の broadcast 配送が 0% になり
  全テスト green のまま気づかれなかった — の再発防止)
- セル単位の鮮度管理(変更 → 影響セルのみ再測 → report 該当セルのみ差し替え)は
  Phase 3 完了後に導入判断

## 5. 未解決の異常(計画と並行)

- **magiconion の wired 2 anchor が基準比で持続的に遅い** — **切り分け済み、
  実回帰ではない**(ledger #16 参照)。video c74 は単一ブロック基準が N=3
  median 容量(66)より上に probe 点を置いていた基準側の問題(#15 の実例)、
  br c101 は budget 際セルのセッション相関 flap。環境要因(.NET 更新・
  同居ゲームサーバー)は時系列で棄却。Phase 2 の基準取り直しで解消する
