# ADR-0004 pilot — confirmatory protocol 凍結のための予備測定

ADR-0004 §3 の pilot を ADR-0005 の凍結 fleet(c8g.16xlarge spot)で実行し、
次を実測して confirmatory protocol の凍結値(block 反復数 N、stopping rule、
drift 許容幅)を導出する:

1. warmup 収束(enet 系 throttle 過渡 >= 15 s を含むか — steady_warmup 実測)
2. block 間分散(同一 cell の seed 違い 3 block)
3. 境界の再現幅
4. 1 block の所要時間(ADR-0005 phase 予算の検証)

## Pre-registration(launch 前に固定)

- fleet: c8g.16xlarge spot × 4(us-west-2、quota 300 内。512 増枠は追わない —
  2026-07-23 owner 決定)

  - `auth1000`: enet × authoritative-state state-1000、conns 4–1024(2 倍刻み)
  - `auth4000`: enet × authoritative-state state-4000、conns 4–512

  enet は steady_min_warmup 15 s(throttle 過渡の下限、ramp-equivalence の前例)。

  pilot data は比較表に混ぜない(ADR-0004 §3)

## 理論包絡(launch 後・結果到着前に固定。2026-07-23 owner 指示)

preset パラメータと netem 条件だけから導いた期待値。実測がここから外れたら
「原因調査」の対象(計測器バグ検出器を兼ねる)。仮定は各行に明記する。

### envbase(raw_udp、client→server 片方向 100 Hz / 64 B)

| 量 | 理論値 | 導出 |
|---|---|---|
| delivery ratio | ≈ 0.990 | 片方向が loss 1% を 1 回通過、再送なし。SLO 0.95 に対し余裕 |
| staleness p99 | ≈ 35–55 ms | 転送 25 ms + 周期 10 ms + loss 起因の 10 ms 欠落(確率 1%)。SLO 300 ms に対し余裕 |
| 負荷 | 線形 N×100 pps | c1024 で 102 k pps・6.6 MB/s — 2 vCPU の raw_udp server には軽い |
| 境界 | **上限 c1024 でも破断しない見込み**(≥1024 で censored 読み) | この cell の目的は境界でなく block 所要時間と環境分散の採取 |

### auth1000(enet、state 1000 B × 20 Hz + input 30 Hz + MD 双方向 10 Hz)

| 量 | 理論値 | 導出 |
|---|---|---|
| server egress | 20 KB/s × N(c1024 で 20.5 MB/s ≈ 164 Mbps) | 1000 B は MTU 内 = 1 packet/client/tick、egress pps ≈ 20 N |
| server ingress | ≈ 40 N pps | LT 30 + MD 10 |
| MD deadline hit | > 0.99 | RTT 50 ms、deadline 200 ms 内に再送機会 ≥2、loss 1% |
| 境界 | c512–1024 帯か、**c1024 censored の可能性あり**(事前開示) | 2 vCPU の per-packet 処理 ~60–80 k pps が律速仮定 |

### auth4000(enet、state 4000 B — MTU 超 3 fragment)

| 量 | 理論値 | 導出 |
|---|---|---|
| server egress | 80 KB/s × N(c512 で 41 MB/s ≈ 328 Mbps)、pps ≈ 60 N | 4000/1370 → 3 fragment |
| LT delivery | fragment 全着が条件: (0.99)³ ≈ 0.970 | 再送なしなら。enet reliable 再送に載る場合は ≈1 で staleness 側へ転嫁 |
| 境界 | auth1000 より下(fragment 3 倍の pps・組み立てコスト) | c256–512 帯の仮説 |

### room(enet、publish 20 Hz / 128 B、O(N²) fanout)

| 量 | 理論値 | 導出 |
|---|---|---|
| server egress bytes | ≈ N(N−1)×20×150 B(c128 で ≈49 MB/s、c256 で ≈196 MB/s) | coalesce 前提でも bytes は不変、frame 数のみ減る |
| 境界 | c64–192 帯(conns 範囲内に収まる見込み) | wired-v3 の enet r20p128=191(lan・budget 違い)を上限側の参照に、2 vCPU + wan で下振れ仮定 |

- 分散(block 間・境界 flap 幅)と warmup 収束時間は理論で先取りできない —
  それを実測するのが本 pilot。理論で先取りできるのは上の SLO 床と負荷包絡のみ
- auth 系 2 cell が理論どおり censored になった場合、境界再現幅は room cell
  からのみ得られる。その場合 conns 上限の引き上げは次 campaign の
  pre-registration で行う(本 campaign 中の変更はしない)

### 理論との突き合わせ

- **一致**: 低負荷点の SLO 床は理論どおり(c4 で LT delivery 0.991 ≈ 理論 0.990、
  staleness p99 55 ms = 理論帯 35–55 ms 上端、MD deadline hit 0.999)。
  netem 注入と計測器の会計は健全
- **乖離 1(envbase)**: c1024 まで壊れない予想に対し 612–653 で破断。
  ただし破断シグナルは 6 packet の server rcvbuf micro-drop — 61 k pps で
  2 vCPU が飽和したのではなく、drop-delta gate の閾値ゼロが際を切っている疑い
- **乖離 2(auth1000)**: 律速仮説(60–80 k pps)以前に、eventual=1.0 という
  「1 packet も落とせない」SLO が有限 drain と組むと確率破断になり、
  seed 間で 60↔406 に暴れる。境界の再現幅がそもそも定義できない

### warmup 収束(pilot 目的 1)

- enet: 全 run で steady 到達 15.5 s = 15 s 床の直後。throttle 過渡 ≥15 s の
  仮定(ledger #14)は fleet でも成立、warmup 25 s cap で十分
- raw_udp: steady 2.75 s。cap 25 s は大きな余裕

### 判定 — protocol 凍結は保留

block 反復数 N・stopping rule・drift 許容幅の凍結は **ledger #27(MD eventual
の knife-edge)と #28(切断の会計)の解決が前提**。auth 系の「境界」が確率事象で
ある間は境界再現幅の統計に意味がなく、ここで凍結すると壊れた値が preset hash に
焼き付く。envbase の分散(全幅 6.5%)は drift 幅導出の材料として有効なので流用可。
再 pilot の対象は auth 系 2 cell(#27 修正で境界が負荷由来になる)と room cell
(#28 修正で c256 の切断が INVALID/censored でなく FAIL になり、本物の境界と
再現幅が取れる)。auth4000 は enet の payload 上限を上げない限り UNSUPPORTED の
まま(ledger #30)。

### 凍結提案(ADR-0004 §3 の手順に基づく導出)

1. **block 反復数 N = 3**(初期提案どおり)。room の実測全幅 5.6% は
   MDE 10% の半分近傍で、flap が IQR を超えた cell のみ 5 へ拡張する条項で足りる
2. **stopping rule = 連続 3 block の median 変動 ≤ 5%、上限 5 block、
   未達は INCONCLUSIVE**(初期提案どおり。room 3 block は median 165、
   変動 (174−164)/165 = 6.1% → この cell は N=5 相当の例になる)
3. **前後 baseline drift 許容幅 = ±5%**。導出: A/A のホスト内反復
   (raw_udp room-lan、ホスト内 ±1–3 / 143 ≈ ±2%)と境界 flap ±1 点を包含する
   最小の丸め幅。anchor gate(fleet median ±10%)より厳しい側に置き、
   drift INVALID は block 単位で発火させる
4. auth 系 scenario の conns 探索上限は 2048 以上に引き上げて confirmatory を
   設計する(≥1024 で range censored のため)

**凍結確定(2026-07-23 owner 承認)**。N=3(flap cell のみ 5)/ stopping rule
(連続 3 block ≤5%、上限 5、未達 INCONCLUSIVE)/ drift ±5% / auth 系 conns
上限 ≥2048。preset hash への実装は confirmatory 設計と合わせて行う。
