# 高性能通信ライブラリの実装・ベンチマークチェックリスト

対象読者: 通信ライブラリの実装者とベンチマークアダプタの実装者。
各項目の具体的な根拠は本プロジェクト14ライブラリの調査（[`adapter-audit.md`](adapter-audit.md)）に基づく。
`§X.Y` は [`dev-notes.md`](dev-notes.md) の事例。

---

## 1. ソケットと OS

### 1.1 バッファサイズ

- [ ] `SO_RCVBUF` / `SO_SNDBUF` を明示設定しているか。OS 既定値（Linux ~208KB）では高スループットに不足し、カーネルがパケットをサイレントにドロップする
- [ ] 設定後に `getsockopt` で実値を確認しているか。`net.core.rmem_max` / `wmem_max` の sysctl 上限を超えた値はサイレントにクランプされる
- [ ] 送受信で非対称になっていないか。SO_SNDBUF だけ小さいと高スループット送信でボトルネックになる
- [ ] バッファを大きくしすぎてバッファブロート（キューイング遅延の増大 → 偽 RTO → 不要な再送）を起こしていないか
- [ ] ライブラリが内部で極端な値を要求していないか。setsockopt に巨大な値を渡してもカーネルがクランプし、意図した値にならない

### 1.2 I/O システムコール

- [ ] 高 PPS で `sendto`/`recvfrom` の 1 パケット 1 syscall が律速になっていないか。`sendmmsg`/`recvmmsg` でバッチ化すると syscall あたりの amortized cost を大幅に下げられる
- [ ] ACK を個別パケットとして送っていないか。reliable パケットごとに専用 ACK パケットを sendto すると syscall 数が 2 倍になる。ピギーバック（データパケットに ACK を相乗り）で削減できる
- [ ] GSO/GRO（Generic Segmentation/Receive Offload）の活用を検討したか。QUIC 系では特に有効

### 1.3 fd とリソース上限

- [ ] conn ごとに socket を作る設計で fd 上限（`ulimit -n`）に引っかからないか。1000 conn で既定 1024 を超えうる
- [ ] per-conn socket は高 conns で syscall 数が O(conns)/tick になる。単一 fd 多重化が可能ならそちらが有利（§5.8）
- [ ] サンドボックス / コンテナ / systemd-run 環境では対話シェルと fd 上限が異なる（§1.7）

### 1.4 MTU とフラグメンテーション

- [ ] ペイロード + ヘッダが Path MTU（Ethernet: 1500 − IP 20 − UDP 8 = 1472B）内に収まるか
- [ ] フラグメンテーション実装がある場合:
  - out-of-order fragment 再組立をサポートしているか。in-order のみだと multi-fragment message で HoL blocking が発生する
  - fragmented unreliable message を暗黙に reliable に昇格していないか。意図しない信頼性保証とオーバーヘッドが加わる
  - 1 fragment でもロスすると全体再送になる IP フラグメンテーションに頼っていないか。アプリ層分割が推奨

## 2. 再送と信頼性

### 2.1 RTO 計算

- [ ] RTO が適応的（RTT 測定に基づく）か固定か。固定 RTO は実装が簡単だが、RTT が RTO より長い経路で偽再送が頻発し、短い経路で不必要に遅延する
- [ ] RTT を測定して SRTT を計算しているのに RTO 計算に使っていないケースがないか（status 表示専用になっていないか）
- [ ] RTO のバックオフ（再送ごとに RTO を増加）を実装しているか。バックオフなしの固定 RTO は輻輳時に再送洪水を起こす
- [ ] RTT を一切測定しない固定 RTO のライブラリで、想定外の高 RTT 経路で偽再送が爆発しないか

### 2.2 Fast Retransmit と ACK

- [ ] Fast Retransmit（重複 ACK やギャップ検出による早期再送）を実装しているか。RTO 待ちのみだとレイテンシ tail が跳ねる
- [ ] ACK 形式の選択: cumulative ACK / selective ACK(SACK bitmap) / range-based ACK / NACK。SACK は帯域効率が良いがヘッダが大きい
- [ ] ACK がピギーバック（データパケットに相乗り）されるか、専用パケットか。専用パケットは ACK traffic のオーバーヘッドが大きい
- [ ] 遅延 ACK のタイミングは適切か。遅延が大きいとスループットは上がるがレイテンシが増える

### 2.3 再送上限とコネクション死活

- [ ] 再送の上限（最大回数 or 最大時間）を設けているか。無制限再送は障害時にリソースを消費し続ける
- [ ] コネクション死活検知（keepalive / timeout）を実装しているか。なければ crashed peer に対して無限に再送し続け、リソースがリークする
- [ ] ライブラリが dead link を検出しても、adapter がその状態フラグを確認しているか。安全装置が無視されていないか

## 3. 輻輳制御

- [ ] 輻輳制御の有無とアルゴリズムを把握しているか。ゲーム用途（固定レート送信）なら CC なしが適切な場合もある
- [ ] ランダムロス環境で CC が過度にレート抑制しないか。loss-based CC（Cubic 等）はランダムロスを輻輳と誤認して cwnd を絞る。BBR 等の BDP-based CC やレート制御ベースの方がランダムロスに強い（§1.9）
- [ ] CC を無効化している場合、ネットワークを溢れさせるリスクを認識しているか
- [ ] loss 時の応答が適切か。cwnd を 1 segment にリセットする Tahoe 動作は回復が極めて遅い。Reno（halving）や BBR（BDP-based recovery）の方が実用的
- [ ] ベンチマーク用に adapter が CC を上書き（無効化、アルゴリズム変更）している場合、その影響を結果の解釈に反映しているか
- [ ] 初期ウィンドウが適切か。大きすぎるとバースト、小さすぎるとスロースタートが長い。slow start なしでフルレートから開始する設計もあるが、バースト耐性に注意

---

## 4. シーケンス番号

- [ ] シーケンス番号のビット幅がユースケースに対して十分か。16-bit は 100K PPS で 0.65 秒、15-bit は 0.33 秒で一周する。高レートの長時間運用では 32-bit 以上が必要
- [ ] ラップアラウンド処理が正しく実装されているか。plain `>` comparison はラップアラウンド時に破綻する。half-space 比較や signed cast が必要
- [ ] ラップアラウンド処理が未実装のライブラリで、想定される PPS × 稼働時間で一周しないか見積もっているか
- [ ] seq 幅の型と比較関数の型が一致しているか（例: 24-bit seq に対して 32-bit halfSpan を使うと誤判定する可能性）

---

## 5. 内部キューとフロー制御

- [ ] 送信キューが満杯になったときの挙動を把握しているか。エラー返却（backpressure）/ ブロック / コネクション切断 のどれか。切断する設計は高負荷で意図しない切断を招く
- [ ] 受信側に受信ウィンドウ（flow control） を実装しているか。なければ sender が receiver のバッファを溢れさせる
- [ ] in-flight の上限（再送バッファのサイズ）がスループットのボトルネックになっていないか。小さすぎると高 BDP 環境でパイプラインを埋められない
- [ ] ゼロウィンドウ時のプローブで再開を検出できるか

---

## 6. スレッドモデル

- [ ] ライブラリのスレッドモデル（single-threaded / 内部 worker / conn ごとにスレッド生成）を把握しているか
- [ ] conn ごとにスレッドを生成する設計で、高 conns でスレッド爆発しないか。1000 conn × 2 threads = 2000 threads でスケジューラが飽和する
- [ ] 「user-threaded mode」で内部スレッドを無効化しても、実際にはスレッドが生成されるバグがないか。Shutdown 時の use-after-free に注意
- [ ] ライブラリのロック粒度（グローバル / per-multiplexer / per-connection）を把握しているか。グローバルロックは高 conns で競合が支配的になる
- [ ] adapter 側のロック設計がライブラリの worker thread 数に対してボトルネックになっていないか（例: 単一 mutex で全 callback を直列化）

---

## 7. メモリアロケーション

- [ ] 送受信のホットパスで heap allocation を避けているか。パケットごとの malloc は高 PPS でアロケータがボトルネックになる
- [ ] アロケーション回避の手法を検討しているか:
  - 事前確保（全メモリをエンドポイント作成時に確保、ゼロアロケーション）
  - プール/スラブ （size-segregated free-list、TLSF、malo slab 等）
  - スタックバッファ （固定サイズの send/recv バッファ）
- [ ] per-conn メモリ使用量を見積もっているか。dedup window + 再送バッファ + 受信順序バッファの合計が conn 数に比例する
- [ ] マネージド言語（C#/Java/Go）では GC ポーズがレイテンシ p99 に影響しないか。packet pool の auto-scale、ArrayPool、off-heap buffer 等で GC 圧力を軽減しているか

---

## 8. 暗号

- [ ] 暗号のオン/オフを選択できるか把握しているか。QUIC 系は TLS 必須、一部のゲームライブラリも暗号必須
- [ ] per-packet 暗号が CPU を支配しないか。暗号有効時は crypto が server CPU のボトルネックになりうる（§1.8）
- [ ] 暗号ありのライブラリを比較する場合、暗号込みの性能として正しく評価しているか（§5.7）
- [ ] 暗号のバッチ処理（AES-NI、ChaCha20 SIMD）を活用できるか

---

## 9. コネクション管理

- [ ] `connect()` がブロッキングの場合、高 conns で接続フェーズだけで秒単位になることを把握しているか（§1.9: conn × RTT の積算）
- [ ] `close()` が内部 worker と競合して deadlock / double-free しないか。同期 teardown が worker と競合する場合は no-op close + プロセス終了か、graceful close のタイムアウトが必要（§5.11）
- [ ] 接続 ID の衝突を処理しているか。時刻ベースの ID 生成は同時接続で衝突する

---

## 10. ベンチマーク計測の落とし穴

### 10.1 計測アーティファクト

- [ ] mixed traffic のレイテンシをチャネル別ヒストグラムで記録しているか。combined histogram では reliable の retx tail が unreliable の p99 に混入し偽の結論を導く（§1.1）
- [ ] client/server の計測窓が重なっているか。ramp-up を server 寿命に含めないと一定割合の delivery が構造的に欠損する（§1.7）
- [ ] warmup データが計測に混入しないか（§5.10: measurement bit）
- [ ] busy-spin idle で CPU 100% を飽和と誤読していないか（§1.6）

### 10.2 統計

- [ ] 結論に使う数字は N ≥ 3 で中央値 + IQR か（§1.5: N=1 は ±10〜20% 振れる）
- [ ] Coordinated Omission を認識しているか。固定間隔送信 + レスポンス待ちだと遅延リクエストの後続が待ち行列に入らず p99 が過小評価される
- [ ] delivery ratio の定義（往復 echo vs 片道配信率）を明記しているか（§5.1）

### 10.3 CPU とメモリ

- [ ] シングルスレッド lib（上限 ~100%）とマルチスレッド lib（SMT2 で ~200%）をスレッドモデル併記で比較しているか（§5.2）
- [ ] ハーネスのメモリオーバーヘッドを分離して server 側 RSS で評価しているか（§5.4）

### 10.4 ネットワークエミュレーション

- [ ] netem `limit` を十分大きく設定しているか（既定 1000 → サイレントドロップ。§1.6）
- [ ] loopback netem は send/recv 両方を通る（片道 25ms → RTT ~50ms、loss 1% → 実効 ~2%）

## 11. 負荷生成の品質

- [ ] 負荷生成器が意図したレートを出し切れているか検証する仕組みがあるか（§1.9）
- [ ] pacing 精度閾値を送信間隔の比率で設定しているか（§1.4: `max(100μs, interval/10)`）
- [ ] multi-process 生成で各プロセスの計測窓が揃っているか（§1.7）

## 12. 診断パターン

| 症状 | まず疑うこと | 参照 |
|------|-------------|------|
| c1 から delivery が低い | adapter/harness バグ（API 誤用、窓ズレ） | §1.9 |
| delivery が `(duration−X)/duration` | client/server の active window ズレ | §1.7 |
| 手動では動くが CI で落ちる | fd 上限、CWD、権限の環境差 | §1.7 |
| CPU 100% だが delivery は低い | busy-spin idle の誤読 or tick 洪水 | §1.6, §1.9 |
| バッファ増で性能悪化 | バッファブロート → 偽 RTO 再送 | — |
| 高 conns で突然 invalid | 負荷生成側の律速 | §1.9 |
| 長時間稼働で突然壊れる | シーケンス番号ラップアラウンド | — |
